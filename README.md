# Індивідуальний проєкт: Flashcard Learner
**Виконав:** студент групи КН-3ХФ Черкасов Ілля
**Практична робота №21.** Вибір теми та постановка задачі. Архітектура застосунку. Визначення MVP.

---

### 1. Тема застосунку
Застосунок дозволяє користувачу створювати власні набори флеш-карток для вивчення іноземних слів чи термінів та перевіряти свої знання через інтерактивне тестування.

### 2. Постановка задачі
* **Цільовий користувач:** Студенти, школярі або будь-хто, хто самостійно вивчає іноземні мови та нову термінологію.
* **Проблема:** Користувачеві важко запам'ятовувати великі обсяги нових слів без структурованого підходу та регулярного повторення. Застосунок надає зручний інструмент для створення персональних наборів карток та швидкої самоперевірки.
* **Очікуваний результат:** Організована локальна база знань, яка дозволяє ефективно тренувати пам'ять у будь-який час.
* **Базові обмеження:** Десктопний застосунок (Windows), працює повністю в офлайн-режимі, дані зберігаються локально на комп'ютері у форматі JSON.

### 3. Сутності та дані
Застосунок працює з двома основними об'єктами:
1. **Набір (Deck)**
   * Ідентифікація: унікальний рядок (UUID).
   * Обов'язкові поля: Назва набору (наприклад, "IT Terms").
   * Опційні поля: Опис набору, дата створення.
2. **Картка (Flashcard)**
   * Ідентифікація: унікальний рядок (UUID).
   * Обов'язкові поля: ID набору (до якого належить картка), Лицьова сторона (Слово), Зворотна сторона (Переклад/Визначення).
   * Опційні поля: Статус вивчення (нове, повторити, вивчено).

### 4. Список екранів застосунку
1. **Головний екран (Dashboard Screen)**
   * *Призначення:* Перегляд списку всіх створених наборів карток.
   * *Елементи UI:* `QListWidget` для списку наборів, кнопки "Створити новий набір", "Видалити набір", "Почати вчити".
   * *Дані та дії:* Показує назви наборів та кількість карток у них. Дозволяє обрати набір для тестування або редагування.
2. **Екран редактора набору (Deck Editor Dialog)**
   * *Призначення:* Додавання, зміна або видалення карток у вибраному наборі.
   * *Елементи UI:* `QLineEdit` для назви набору, `QTableWidget` для списку слів, кнопки "Додати слово", "Видалити слово", "Зберегти".
   * *Дані та дії:* Відображає таблицю "Слово — Переклад". Дозволяє вводити та редагувати текст.
3. **Екран тестування (Study Window)**
   * *Призначення:* Інтерактивна сесія перевірки знань.
   * *Елементи UI:* Великий `QLabel` по центру для тексту картки, кнопка "Показати відповідь", кнопки оцінки "Знаю" та "Не знаю".
   * *Дані та дії:* Показує слово з картки. Після кліку на "Показати відповідь" відображає переклад. Користувач оцінює себе, і програма перемикає на наступну картку.

### 5. Основні сценарії користувача (User Flows)
1. **Стартовий сценарій (Проходження тесту):** Відкрив застосунок ➔ вибрав набір "Англійська В1" ➔ натиснув "Почати вчити" ➔ побачив слово "Object" ➔ натиснув "Показати відповідь" ➔ побачив "Об'єкт" ➔ натиснув "Знаю" ➔ перейшов до наступного слова ➔ отримав вікно з фінішем сесії.
2. **Сценарій створення набору:** Натиснув "Створити новий набір" ➔ ввів назву "Німецька А1" ➔ натиснув "Додати слово" ➔ ввів "Der Hund / Собака" ➔ натиснув "Зберегти" ➔ набір з'явився у списку на Головному екрані.
3. **Сценарій редагування:** Вибрав набір ➔ натиснув "Редагувати" ➔ змінив помилку в слові ➔ натиснув "Зберегти" ➔ оновлені дані записалися у файл.

### 6. Архітектура застосунку (Шари відповідальності)
* **UI (Інтерфейс):** Класи вікон та віджетів. Відповідають лише за відображення та збір кліків.
  * *Приклади класів:* `MainWindow`, `DeckEditorDialog`, `StudyWindow`.
* **Логіка (Business Logic):** Правила роботи застосунку.
  * *Приклади класів:* `StudySessionManager` (керує чергою карток під час тесту), `DeckValidator` (перевіряє, чи не порожні поля).
* **Дані (Data Layer):** Збереження та завантаження.
  * *Приклади класів:* `JsonRepository` (запис/читання JSON файлу), `FlashcardModel` (структури даних в C++).

### 7. Визначення MVP (Minimum Viable Product)
* **Ключовий end-to-end сценарій:** Створення набору з кількома картками, збереження його на диск та успішне проходження тестування від першої до останньої картки.
* **Мінімальний функціонал (Must-have):**
  1. Створення, редагування назви та видалення набору.
  2. Додавання та видалення карток всередині набору.
  3. Режим тестування (показ картки ➔ показ відповіді за кнопкою).
  4. Автоматичне збереження та завантаження даних з JSON-файлу.
* **Після MVP (Backlog):**
  1. Алгоритм інтервальних повторень (Spaced Repetition).
  2. Статистика та графіки прогресу вивчення.
  3. Імпорт/Експорт карток через CSV-файли.
  4. Можливість додавати аудіо-озвучку для слів.
  5. Темна тема для інтерфейсу.

### 8. Критерії готовності MVP
* Присутні та працюють усі 3 головні екрани.
* Ключовий сценарій "Створити ➔ Наповнити ➔ Пройти тест" працює стабільно і без вильотів програми.
* Після перезапуску програми всі створені раніше картки успішно завантажуються з файлу.

## Практична робота №22. Проєктування інтерфейсу: екрани, навігація та стани UI

### 1. Детальний список та призначення екранів MVP
Застосунок містить 3 ключові екрани, кожен з яких вирішує чітку підзадачу користувача[cite: 2]:

1. **Головний екран (Dashboard Screen)**[cite: 2]
   * *Призначення:* Менеджмент наборів карток та вибір контексту роботи[cite: 2].
   * *Дані, що відображаються:* Список усіх наявних наборів (`Назва набору`, `Кількість карток у ньому`)[cite: 2].
   * *Доступні дії (Команди):* `Створити набір`, `Редагувати набір`, `Видалити набір`, `Почати навчання`[cite: 2].

2. **Екран редактора набору (Deck Editor Dialog)**[cite: 2]
   * *Призначення:* Модифікація вмісту конкретного набору карток[cite: 2].
   * *Дані, що відображаються:* Назва набору, таблиця слів (`Слово/Термін` — `Переклад/Визначення`)[cite: 2].
   * *Доступні дії (Команди):* `Додати рядок`, `Видалити рядок`, `Зберегти`, `Скасувати`[cite: 2].

3. **Екран тестування (Study Window)**[cite: 2]
   * *Призначення:* Інтерактивна перевірка знань методом самооцінювання[cite: 2].
   * *Дані, що відображаються:* Прогрес сесії (`Картка X із Y`), поточний текст (слово або переклад), фінальний результат[cite: 2].
   * *Доступні дії (Команди):* `Показати відповідь`, `Знаю` / `Не знаю`, `Перервати тест`[cite: 2].

### 2. Структура екранів (Каркас UI)
* **Головний екран:** Базується на `QMainWindow`[cite: 2]. Центральна область — вертикальний макет (`QVBoxLayout`), де зверху розміщено заголовок, по центру великий `QListWidget` для відображення колекцій, а знизу — горизонтальна панель команд `QHBoxLayout` з кнопками управління[cite: 2].
* **Екран редактора:** Реалізований як `QDialog`[cite: 2]. Компонування сіткою (`QGridLayout`)[cite: 2]. Зверху поле `QLineEdit` для назви[cite: 2]. Центральна частина — `QTableWidget` на 2 колонки[cite: 2]. Справа — вертикальний стовпчик кнопок для маніпуляцій з рядками, знизу — кнопки збереження[cite: 2].
* **Екран тестування:** Реалізований як окреме вікно `QWidget`[cite: 2]. Центральна область — великий інформер `QLabel` з рамкою для картки (шрифт збільшений для читабельності)[cite: 2]. Знизу — динамічний блок кнопок, що змінюється залежно від кроку[cite: 2].

### 3. Стани екранів
Кожен екран має щонайменше два стани відображення залежно від умов[cite: 2]:

* **Головний екран:**[cite: 2]
  * *Порожній стан (Перший запуск):* Список наборів порожній[cite: 2]. По центру відображається підказка: "У вас немає створених наборів. Натисніть 'Створити', щоб додати перший"[cite: 2]. Кнопки "Редагувати", "Видалити", "Почати навчання" заблоковані (`disabled`)[cite: 2].
  * *Звичайний стан:* Список заповнений[cite: 2]. При виділенні будь-якого набору всі кнопки управління стають активними[cite: 2].
* **Екран редактора:**[cite: 2]
  * *Режим редагування (Звичайний):* Користувач вводить текст[cite: 2]. Кнопки "Зберегти" та "Скасувати" активні[cite: 2].
  * *Стан помилки валідації:* Виникає, якщо назва набору порожня або в таблиці є незаповнені клітинки[cite: 2]. При спробі збереження виводиться повідомлення `QMessageBox`, а збереження блокується до виправлення[cite: 2].
* **Екран тестування:**[cite: 2]
  * *Стан "Питання":* Показується тільки лицьова сторона картки[cite: 2]. Активна лише кнопка "Показати відповідь"[cite: 2]. Кнопки оцінки "Знаю/Не знаю" приховані (`hidden`)[cite: 2].
  * *Стан "Відповідь":* Текст картки змінюється на переклад[cite: 2]. Кнопка "Показати відповідь" хоється, стають видимими кнопки "Знаю" та "Не знаю"[cite: 2].
  * *Стан "Фініш":* Картки закінчились[cite: 2]. Текст змінюється на "Вітаємо! Тест завершено"[cite: 2]. Кнопки управління ховаються, з'являється одна кнопка "Повернутись на головну"[cite: 2].

### 4. Навігаційна схема (Граф переходів)
* **Стартовий пункт:** Головний екран (Dashboard)[cite: 2].
* **Переходи:**[cite: 2]
  1. `Головний екран` ➔ `Екран редактора (Новий)` | *Тригер:* Клік на кнопку "Створити набір"[cite: 2].
  2. `Головний екран` ➔ `Екран редактора (Існуючий)` | *Тригер:* Виділення набору + кнопка "Редагувати"[cite: 2].
  3. `Екран редактора` ➔ `Головний екран` | *Тригер:* Клік на "Зберегти" або "Скасувати"[cite: 2].
  4. `Головний екран` ➔ `Екран тестування` | *Тригер:* Виділення набору + кнопка "Почати навчання"[cite: 2].
  5. `Екран тестування` ➔ `Головний екран` | *Тригер:* Клік на "Перервати тест" або завершення сесії[cite: 2].

### 5. Обґрунтування модальності вікон
* **Редактор набору карток** реалізується як **модальний діалог** (`QDialog::exec()`)[cite: 2]. Це критично, оскільки під час модифікації слів користувач не повинен мати можливості клікати на Головне вікно, щоб уникнути конфліктів у збереженні JSON файлу[cite: 2].
* **Екран тестування** реалізується як **немодальне вікно**[cite: 2]. Воно повністю закриває або заміщує робочий простір, оскільки сесія навчання є головним фокусом користувача[cite: 2].

## Практична робота №23. Проєктування даних: сутності, модель стану та збереження

### 1. Модель даних (Сутності)
Застосунок базується на ієрархічній структурі даних, де `Deck` (Набір) є контейнером для `Flashcard` (Картки).

* **Entity: Deck (Набір)**
  * `id` (String, UUID): Унікальний ідентифікатор.
  * `title` (String): Назва набору.
  * `created_at` (Timestamp): Дата створення.
  * `cards` (List): Список об'єктів типу Flashcard.

* **Entity: Flashcard (Картка)**
  * `id` (String, UUID): Унікальний ідентифікатор.
  * `front` (String): Лицьова сторона (термін).
  * `back` (String): Зворотна сторона (визначення/переклад).
  * `difficulty` (Int): Поточний рейтинг складності (наприклад, 1-5).

### 2. Модель стану застосунку (Application State)
Це дані, які застосунок тримає в оперативній пам'яті під час виконання, але не обов'язково зберігає постійно:

* **Global State:**
  * `current_deck`: ID набору, який зараз редагується або вивчається.
  * `is_dirty`: Прапорець, що сигналізує про незбережені зміни в поточному наборі.
* **Study Session State (Стан тестування):**
  * `session_deck`: Копія набору для вивчення.
  * `current_card_index`: Індекс поточної картки в сесії.
  * `is_showing_answer`: Булева змінна (Чи бачить користувач переклад зараз?).

### 3. Формат збереження даних (JSON)
Дані зберігаються в локальному файлі `data.json` у такому форматі:

```json
{
  "decks": [
    {
      "id": "deck-123",
      "title": "IT Terms",
      "cards": [
        {
          "id": "card-001",
          "front": "Array",
          "back": "Масив",
          "difficulty": 3
        }
      ]
    }
  ]
}
